# Phase 09 — Deterministic explicit routing

## Expected result

- Explicit model and effort commands are recognized before agent sampling.
- Model resolution uses only the Codex App Server catalog and rejects hidden, ambiguous or unsupported profiles.
- The explicit path performs no CaMe MCP call, state preflight, agent-generated checkpoint or context compaction.
- Autonomous MCP routing accepts only `model` and `effort`; reason and continuation are generated by trusted server code.
- The active Codex thread is preserved and receives the selected profile.

## Production adversarial review

### Cycle 1

- Added a bounded parser for Italian and English imperative profile commands.
- Required the command to be the only turn input and rejected malformed or composite requests before App Server forwarding.
- Added strict model-query and effort bounds plus a fixed effort allowlist.

### Cycle 2

- Moved explicit profile application into the serialized handoff queue.
- Resolved model aliases against the App Server catalog, excluding hidden, ambiguous, unsupported and unsafe profiles.
- Cached the validated catalog for the session so the first explicit change performs model discovery and later changes require only the settings update.
- Preserved the active `threadId` and rejected cross-thread or in-turn changes.

### Cycle 3

- Reduced `came_switch_model` input to strict `model` and `effort` fields.
- Rejected extra agent-controlled routing context before IPC execution.
- Replaced agent-generated reason and continuation with fixed server-owned values.
- Recorded explicit changes without raw user or agent text in the audit log.

### Cycle 4

- The first real TUI run exposed that `turn/start.collaborationMode.settings` takes precedence over top-level model and effort.
- Rewrote both protocol locations while preserving unrelated collaboration settings.
- Added a regression assertion for `collaborationMode` precedence and repeated the focused runtime, handoff and integration tests.

### Cycle 5

- Re-read the complete production, test and plugin diff for prompt injection, unsafe catalog data, thread divergence, stale handoff state and implicit fallback behavior.
- Confirmed that the fixed acknowledgement contains only validated canonical profile tokens.
- No actionable findings remained.

## Test evidence

### Deterministic behavior

- Explicit commands are applied before the forwarded `turn/start`.
- Invalid, composite, cross-thread, in-turn, ambiguous, unsupported and unsafe requests fail closed.
- A rejected settings update moves the router to the terminal failed state.
- A second explicit switch reuses the cached model catalog.
- Autonomous MCP input with `reason` or `continuation` is rejected by the strict tool schema.
